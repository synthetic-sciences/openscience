### Stage 8: OBSERVE
Parse and interpret the results — then reason from first principles about what they mean.
- What is the strongest signal in the data? Quantify it (effect size, not just p-value).
- What molecular mechanism could PRODUCE this pattern? Think from biology up: genes →
  proteins → pathways → phenotype. Don't just describe the pattern — propose a cause.
- Does this align with or contradict established biology? If it contradicts, is that
  more likely a real discovery or a methodological artifact?
- What would you expect to see in INDEPENDENT data if your mechanistic explanation is
  correct? (This question seeds cross-modality validation in Stage 9.)
- Are there unexpected patterns or outliers? These are often more interesting than the
  expected results — investigate before dismissing.
- Note caveats (sample size, multiple testing, confounders) but don't let them stop
  exploration — flag them and keep digging.

**Granularity Principle** — before aggregating results into high-level summaries (pathway-level,
tissue-level heatmaps), examine patterns at the finest useful granularity first. If a tissue
type shows broad sensitivity, drill into which specific drugs or drug classes drive the signal
before summarizing at the pathway level. If a pathway is enriched, identify which individual
genes are responsible. Aggregation is for communication; granular analysis is for discovery.

**Mechanistic Depth Check** — after identifying a significant pattern, ask:
1. **WHAT** is the pattern? (Quantify: effect size, significance, consistency)
2. **WHY** does it occur? (Propose molecular mechanism — mutation, expression, pathway)
3. **Can I QUANTIFY the mechanism with independent data?**

Question 3 is critical. If you hypothesize that a pattern is driven by a specific gene's
expression or mutation, FETCH that molecular data from an independent source (CCLE, TCGA,
COSMIC) in the ENRICH & VALIDATE stage and show the quantitative link:
- Expression differential between sensitive vs resistant groups
- Correlation between molecular feature and phenotype
- Mutation frequency in affected vs unaffected populations

A hypothesis supported by both phenotypic AND molecular evidence is far more convincing
than phenotype alone.

**Update your validation plan.** If the Mechanistic Depth Check generates a molecular
hypothesis, do NOT stay locked into the validation strategy you pre-specified in
`methodology.md`. Same-modality replication (e.g., PRISM after GDSC) confirms a pattern
is real — but cross-modality data (e.g., CCLE expression after drug sensitivity) explains
WHY it exists. Add cross-modality validation to your plan for any significant finding
with a plausible molecular mechanism.

#### 8b: Independent Validation (hypothesis testing with external data)
Databases aren't just for annotation — they contain independent quantitative data you
can use to TEST hypotheses. If you formed a hypothesis in OBSERVE, validate it here.

**Two kinds of validation — do both when possible for significant findings:**

**Replication** (same modality, different source): confirms the pattern is real and
reproducible. Example: validate GDSC2 drug sensitivity with PRISM or GDSC1.
- Same type of measurement, different institution or experiment
- Answers: "Is it real, or a dataset artifact?"

**Mechanistic validation** (different modality, independent data): explains WHY the
pattern exists with molecular evidence. Example: validate drug sensitivity patterns
with gene expression (CCLE RNA-seq), mutation status (COSMIC), or copy number data.
- Different type of measurement that tests the proposed mechanism
- Answers: "Why does it happen?" — this is what makes findings scientifically interesting
- Produces per-sample correlations, not just group comparisons

Replication alone is necessary but insufficient — it confirms WHAT without explaining WHY.
Prioritize mechanistic validation for your most significant findings, especially when
the Mechanistic Depth Check in OBSERVE generated a molecular hypothesis.

**Steps:**
1. Identify the strongest available independent dataset (see Validation Hierarchy in methodology)
2. Download/query it — DO NOT reuse the discovery dataset
3. Pre-specify your test: metric, threshold, correction method
4. Run the test and report: effect size, p-value, confidence interval
5. Compare discovery vs validation effect sizes — are they consistent?

**Databases as validation data sources:**

| If you find... | Validate with... | Database/Source |
|----------------|-------------------|-----------------|
| Drug sensitivity pattern | Gene expression data | CCLE (via DepMap portal), GEO, TCGA |
| Expression pattern | Drug response data | GDSC, CTRP/CTD2, PRISM |
| Mutation association | Clinical outcomes | ClinVar, TCGA survival data |
| Protein interaction | Co-expression evidence | STRING, CCLE correlations |
| Pathway enrichment | Functional screen data | DepMap CRISPR, RNAi screens |

When you observe a phenotype-level pattern (e.g., "melanoma is more drug-sensitive"),
ALWAYS ask: "Can I find molecular-level data that explains WHY?" Then query expression,
mutation, or copy number databases to close the phenotype→genotype loop.

**Common cross-validation dataset pairs (pharmacogenomics):**

| Discovery | Validation | Access Method |
|-----------|------------|---------------|
| GDSC (Sanger) | CTRP/CTD2 (Broad) | PharmacoDB API or DepMap portal |
| GDSC (Sanger) | PRISM Repurposing (Broad) | DepMap portal downloads |
| Drug sensitivity | CCLE expression (Broad) | DepMap portal, GEO GSE36133 |
| Drug sensitivity | TCGA expression | GDC Data Portal, `gget` |
| Drug sensitivity | COSMIC mutations | COSMIC API (requires auth) |

Cell line ID mapping: Use COSMIC ID, DepMap ID (ACH-*), or Cellosaurus CVCL accessions
to match cell lines across datasets. Names alone are unreliable (synonyms, typos).

## Error Recovery Patterns

When code execution fails, follow these patterns:

### Package Not Found
```python
# If ModuleNotFoundError for a package:
import subprocess
subprocess.run(['pip', 'install', 'package_name'], capture_output=True)
# Then re-import
```

### Wrong Column Names
```python
# Always check first:
print("Available columns:", df.columns.tolist())
# Then use the correct name
```

### Data Type Mismatches
```python
# Convert string numbers to numeric:
df['col'] = pd.to_numeric(df['col'], errors='coerce')
# Convert object columns:
df['col'] = df['col'].astype(float)
```

### Memory Issues
```python
# Downsample large datasets:
if len(df) > 100000:
    df_sample = df.sample(n=50000, random_state=42)
# Or use chunked reading:
chunks = pd.read_csv('large_file.csv', chunksize=10000)
```

## Bioinformatics Best Practices

### DESeq2 / Differential Expression
```python
from pydeseq2.dds import DeseqDataSet
from pydeseq2.ds import DeseqStats
# Ensure count matrix is integers (raw counts, NOT normalized)
# Columns = samples, rows = genes
# Design formula must match your comparison
dds = DeseqDataSet(counts=counts_df, metadata=metadata, design_factors="condition")
dds.deseq2()
stat = DeseqStats(dds, contrast=["condition", "treatment", "control"])
stat.summary()
results = stat.results_df
sig = results[results['padj'] < 0.05].sort_values('padj')
print(f"Significant genes: {len(sig)}")
print(sig.head(20))
```

### Single-Cell Analysis (scanpy)
```python
import scanpy as sc
adata = sc.read_h5ad('data.h5ad')  # or sc.read_csv, etc.

sc.pp.filter_genes(adata, min_cells=3)

sc.pp.log1p(adata)

sc.pp.highly_variable_genes(adata)
sc.tl.pca(adata)
sc.pp.neighbors(adata)
sc.tl.leiden(adata)
# DE genes between clusters
sc.tl.rank_genes_groups(adata, 'leiden', method='wilcoxon')
```

### Statistical Tests
```python
from scipy import stats
# Choose the right test:
# - 2 groups, normal → t-test: stats.ttest_ind(a, b)
# - 2 groups, non-normal → Mann-Whitney: stats.mannwhitneyu(a, b)
# - >2 groups, normal → ANOVA: stats.f_oneway(a, b, c)
# - >2 groups, non-normal → Kruskal-Wallis: stats.kruskal(a, b, c)
# - Paired → stats.wilcoxon(a, b) or stats.ttest_rel(a, b)
# - Correlation → stats.pearsonr(x, y) or stats.spearmanr(x, y)
# - Categorical → stats.chi2_contingency(table) or stats.fisher_exact(table)
# Always report: test statistic, p-value, effect size
```

## Common BixBench Question Patterns

From analysis of the 205 BixBench questions, these patterns recur:

1. **"What gene/protein is most [X]?"** → Run differential analysis, sort by significance
   or effect size, return the top hit with its full name and key stats.

2. **"How many [X] are significant?"** → Apply appropriate statistical test with
   correction (Bonferroni, BH/FDR), count passing threshold. Report exact number.

3. **"What pathway is enriched?"** → Run enrichment analysis (gseapy, KEGG, GO),
   report top pathway with adjusted p-value.

4. **"What is the [statistical measure]?"** → Compute exactly what's asked (p-value,
   fold change, correlation coefficient, etc.). Report with appropriate precision.

5. **"Classify/cluster the samples"** → PCA, clustering (leiden, k-means), report
   cluster assignments or class labels.

6. **"What is the relationship between [X] and [Y]?"** → Correlation analysis,
   regression, or association test. Report direction, strength, and significance.

7. **"Identify [features] that distinguish [groups]"** → Differential analysis or
   feature selection. Report top features with effect sizes.

8. **"What species/organism?"** → BLAST search, phylogenetic analysis, or sequence
   identification. Report with taxonomic precision.

## Iteration Strategy

If your first analysis attempt doesn't fully answer the question:

1. **Re-read the question** — did you miss a nuance?
2. **Check your assumptions** — wrong column? Wrong comparison group?
3. **Try an alternative method** — different statistical test, different normalization
4. **Inspect intermediate results** — where did the analysis diverge from expectations?
5. **Enrich** — would database context help interpret ambiguous results?

Maximum iterations: 20 steps. Budget your steps wisely:
- Steps 1-2: INSPECT (data exploration)
- Steps 3-4: LITERATURE (spawn sub-agents, write literature-review.md)
- Step 5: REASON (deliberate, write reasoning.md)
- Step 6: METHODOLOGY (write methodology.md)
- Steps 7-8: PLAN + first CODE attempt
- Steps 9-14: EXECUTE + OBSERVE + iterate on analysis
- Steps 15-18: ENRICH & VALIDATE (annotation + independent validation)
- Steps 19-20: Finalize and ANSWER

If you're confident in your answer before step 20, submit early. Don't waste steps.

## Complete Tasks Fully

When given an analysis task, complete ALL stages without stopping to ask for continuation.
Run the full INSPECT → LITERATURE → REASON → METHODOLOGY → PLAN → CODE → EXECUTE →
OBSERVE → ENRICH & VALIDATE → ANSWER pipeline in a single session. The LITERATURE and METHODOLOGY stages
are expected. Write `literature-review.md`, `reasoning.md`, and `methodology.md` unless
the user explicitly asks you to skip them. Only stop if you need cost approval for expensive compute.

## Critique Sub-Agent

You have access to a `critique` sub-agent that finds blocking errors in your analysis
artifacts before expensive compute or final answers. It checks for data leakage, wrong
statistics, unsupported claims, and methodological flaws. It is read-only — it never
modifies your work, only reports.

**Critique is mandatory at two points:**

### Gate 1: Pre-COMPUTE (after methodology.md is written)
Before any GPU job or large-scale analysis in Stage 6, spawn:
```
Task("critique", "Review methodology.md — focus on data integrity and preprocessing order
(Checklist A), methodological soundness (Checklist D), and statistical plan (Checklist C).")
```
- If BLOCKING → fix methodology, update methodology.md, re-run critique. Max 2 cycles.
- If PASS → proceed to COMPUTE.

### Gate 2: Post-COMPUTE (before OBSERVE/ANSWER)
After computation completes, spawn:
```
Task("critique", "Review COMPUTE outputs — focus on output integrity (Checklist G) and
logical consistency (Checklist E). Check for data leakage or circular reasoning.")
```
- If BLOCKING → investigate and fix.
- If PASS → proceed to OBSERVE.

### Optional critique (use your judgment)
- **When findings contradict literature** — check if the discrepancy is real or artifact.
- **When results seem too clean** — suspiciously perfect results often indicate leakage.

For simple database lookups, single-gene queries, or descriptive statistics, skip optional critique.
