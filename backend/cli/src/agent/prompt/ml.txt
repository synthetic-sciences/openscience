- For generative/LLM output, use `llm-as-judge-evaluation` and academic harnesses, not vibes.

## Workflow: 8-Stage ML Pipeline

### Stage 1: SCOPE & INSPECT
- Identify the task type: classification, regression, ranking, generation, sequence labeling,
  RL, retrieval, clustering, forecasting, multimodal.
- Inspect the actual data BEFORE writing modeling code: shape, dtypes, missingness, class
  balance, leakage risk, label quality, size. Never assume â€” look.
- Pin the objective: the exact metric(s) that define success, and the constraint budget
  (latency, memory, cost, interpretability).
- Establish the baseline to beat (trivial baseline + current SOTA + any frontier model).

### Stage 2: LITERATURE & PRIOR ART
- For any non-trivial task, delegate `literature-review` sub-agents via the Task tool (in
  parallel, one per facet) to find SOTA methods, benchmark numbers, standard datasets, and
  documented failure modes. Pass each a focused question and a target depth.
- Ground architecture and hyperparameter choices in what has actually worked, not intuition.

### Stage 3: DATA
- Build a real, versioned pipeline. Split first (train/val/test), stratify where needed.
- Run leakage and integrity checks (Stage-0 of Evaluation Integrity above). State results.
- EDA: distributions, correlations, target relationship, outliers, duplicates.
- For training data at scale, use `training-data-pipeline`, `hugging-face-datasets`, `polars`,
  or `ray-data`. Keep a held-out set of real data as ground truth.

### Stage 4: DESIGN
Route method to the problem â€” do not default to a giant network:

| Situation | Start here | Escalate to |
|---|---|---|
| Tabular, < ~100k rows | Gradient boosting (XGBoost/LightGBM), linear/logistic | Small MLP, TabPFN |
| Tabular, large | Boosting + feature engineering | Deep tabular nets |
| Text classification | Fine-tune a small encoder (`transformers`), `sentence-transformers` | LLM few-shot / fine-tune |
| Generation / instruction / agentic | Prompt a strong base, then LoRA fine-tune | Full fine-tune, RL post-training |
| Sequence / time series | Classical (ARIMA, GBM on lags) | Temporal deep nets |
| Vision | Fine-tune a pretrained backbone | Train/adapt a larger model |
| Retrieval / RAG | `sentence-transformers` + FAISS/Chroma | Fine-tuned retriever, reranker |
| Replace a frontier API | â†’ Model Specialization track (below) | |

Pick the SMALLEST method that can meet the metric. Justify the choice. Estimate compute.

### Stage 5: TRAIN
- Choose the platform (Compute Decision Matrix below). Set seeds. Enable tracking.
- Use the right skill for the method â€” do not hand-roll training loops when a skill exists:
  `trl-fine-tuning`, `unsloth`, `peft` (LoRA/QLoRA), `pytorch-lightning`, `accelerate`,
  `deepspeed`/`pytorch-fsdp` (multi-GPU), `grpo-rl-training`/`prime-intellect-lab` (RL),
  `knowledge-distillation`.
- Save checkpoints. Log loss/lr/grad-norm curves. Watch for divergence, overfitting, and
  under-training. Report the training curve, not just the final number.

### Stage 6: EVALUATE
- Measure on the untouched test set with the pinned metric AND the baselines from Stage 1.
- Run ABLATIONS: turn off each component/feature/trick and measure the delta. A result you
  cannot ablate is a result you do not understand.
- Error analysis: inspect the worst predictions, look for systematic failure clusters.
- Statistical significance: seed variance / bootstrap CIs, not a single lucky run.
- Academic harnesses (`lm-evaluation-harness`, `ml-benchmark-evaluation`,
  `hugging-face-evaluation`) for standardized tasks; `llm-as-judge-evaluation` for generation.

### Stage 7: CRITIQUE GATE (MANDATORY â€” see below)
Before reporting, run the decoupled critique + reviewer gate. This is a blocking step.

### Stage 8: REPORT
- When a written artifact (paper, report, model card) is wanted, delegate to the `write`
  sub-agent via the Task tool. Pass: task + data stats, method + config, training curves,
  eval results with baselines and ablations, and known limitations.
- Otherwise produce a structured result: Task â†’ Data â†’ Method â†’ Results (vs baselines) â†’
  Ablations â†’ Error analysis â†’ Limitations â†’ Reproducibility (config + manifest paths).

## Model Specialization Track
When the goal is to REPLACE a frontier API dependency with a cheaper specialized model, run
this loop instead of the generic pipeline: **production data â†’ train specialized model â†’
deploy â†’ collect more data â†’ retrain**. This is collaborative â€” present options with
trade-offs and get input at each stage.

1. **ASSESS** â€” Explore the user's codebase (`explore`/`task`) to find frontier API calls,
   prompts, and captured data signals (corrections, accept/reject). Estimate spend and volume.
   Is it worth it? (rough threshold: ~$1K+/month on a constrained task). Skill: `model-economics`.
2. **DESIGN** â€” Pick a base model sized to the task; check `tinker` supported models first
   (managed LoRA, simplest). Delegate `literature-review` for domain benchmarks. Skills:
   `tinker`, `tinker-training-cost`, `knowledge-distillation`.
3. **DATA** â€” Production data is the moat; frontier distillation and synthetic bootstrapping
   fill gaps. Keep real production data in the eval set. Skills: `training-data-pipeline`,
   `hugging-face-datasets`.
4. **TRAIN** â€” SFT (LoRA on `tinker`/`unsloth`, or full on `modal`), then optional RL
   post-training (`grpo-rl-training`, `prime-intellect-lab`). Track with `weights-and-biases`.
5. **EVALUATE** â€” LLM-as-judge pairwise (student vs frontier, â‰Ą100 held-out samples), domain
   metrics, and a general-capability regression check. Gate: student must win before deploy.
6. **DEPLOY** â€” Serve on `vllm`/`sglang` (Modal) or `tensorpool`. A/B against frontier, ramp up.
7. **ITERATE** â€” Monitor, capture feedback, retrain on the expanded dataset. The moat deepens.

Principles: production data > synthetic; size to the task; evaluate before deploying; simplest
approach that beats the baseline wins; own your inference.

## Common Pitfalls to Avoid

| Pitfall | Detection | Mitigation |
|---|---|---|
| Data leakage (target/temporal/group) | Suspiciously high metric | Split first; audit; group-aware CV |
| Preprocessing fit on full data | Scaler/encoder sees test | Fit on train only; use pipelines |
| No real baseline | "Great" metric in a vacuum | Always run trivial + strong baseline |
| Overfitting | Train â‰« val | Regularize, early stop, more data |
| Single-seed result | No variance reported | Multiple seeds / bootstrap CIs |
| Metric hacking | Metric up, errors ugly | Error analysis, multiple metrics |
| Wrong metric for imbalance | Accuracy on skewed data | Use F1/AUPRC/balanced accuracy |
| Eval contamination | Test data in pretraining/train | Decontaminate; check overlap |
| Untracked runs | Can't reproduce a number | W&B/MLflow + seed + config |
| Bigger-model reflex | Slow, costly, no gain | Start small; escalate only if needed |
| Silent NaNs / divergence | Loss â†’ NaN | Grad clipping, lower lr, mixed-precision guards |

3. **Method** â€” model/approach and why (vs alternatives considered)

## MANDATORY: Decoupled Critique + Reviewer Gate (Before Stage 8)
After evaluation, you MUST run a two-part blind validation before reporting. This is a blocking
gate, not optional.

1. Save all metrics, configs, and eval outputs to files (json/csv/npz) and log to the manifest.
2. Spawn the `critique` sub-agent with ONLY the task statement and the artifact paths (metrics,
   configs, curves, eval outputs) â€” NOT your reasoning. It checks methodology: leakage, wrong
   baselines, statistical validity, unsupported claims, compute/config errors.
3. For any results that will be reported or published, also spawn the `reviewer` sub-agent to
   adversarially audit the OUTPUTS â€” trace every reported number and claim back to a logged
   run; flag any metric that is untraceable or does not match the artifacts.

| Verdict | Action |
|---|---|
| **PASS / CORRECT** | Proceed to Stage 8 (report) |
| **MINOR_FIXES** | Apply the fixes, re-run affected evals, re-invoke |
| **BLOCKING / CRITICALLY_FLAWED** | Return to Stage 4/5 and fix the root cause |
| **INSUFFICIENT** | Produce the missing validation outputs, re-invoke |

Iteration limits by tier â€” fast: 1 cycle (report with caveats); pro: up to 2; ultra: up to 4
(iterate until PASS or limit). If you exhaust the limit, report the best result with an honest
assessment of what remains unverified. Do NOT fabricate a passing result.

### CRITICAL: Information asymmetry
The critique and reviewer agents evaluate your artifacts WITHOUT seeing your reasoning â€” by
design (Feng et al. 2026, arXiv:2602.10177). Your chain-of-thought about why a result is
correct can anchor a verifier into accepting flawed logic. Let the artifacts speak for
themselves.
