## Workflow: 7-Stage Physics Pipeline

### Stage 1: INSPECT & PARSE
- Identify the physics domain (mechanics, fluids, E&M, quantum, thermodynamics, statistical mechanics, optics, relativity, ...)
- Parse input data (files, equations, parameters, boundary conditions)
- Validate units using `pint` or manual dimensional analysis
- Identify symmetries, conservation laws, and constraints
- Check for known analytical solutions or limiting cases

### Stage 2: LITERATURE & CONTEXT
- Retrieve relevant physical laws, equations, fundamental constants
- Identify applicable approximations:
  - Incompressible vs compressible flow
  - Non-relativistic vs relativistic
  - Classical vs quantum
  - Continuum vs discrete
  - Linear vs nonlinear
  - Steady-state vs transient
- Reference established results (textbook solutions, benchmark problems)

### Stage 3: METHODOLOGY SELECTION
Route to appropriate method based on problem type:

| Problem Type | Primary Method | Fallback |
|---|---|---|
| ODE / initial value problem | `scipy.integrate.solve_ivp` | Symplectic integrator for Hamiltonian systems |
| ODE / boundary value problem | `scipy.integrate.solve_bvp` | Shooting method |
| PDE (elliptic/parabolic/hyperbolic) | FEM (FEniCS/scikit-fem) or spectral | PINN (DeepXDE) |
| Eigenvalue problem | `scipy.linalg.eig` / sparse eigensolver | |
| Equation discovery from data | PySR (symbolic regression) | SINDy (pysindy) |
| Conservation law / symmetry search | Noether analysis | Symbolic regression |
| Phase space / stability analysis | Numerical integration + visualization | |
| Curve fitting / parameter estimation | `scipy.optimize.curve_fit` or `lmfit` | MCMC (emcee) |
| Bayesian inference | `emcee` or `PyMC` | |
| Monte Carlo simulation | Custom Metropolis / Wolff | |
| Spectral / frequency analysis | `scipy.fft` | Welch PSD, wavelets |
| Dimensional analysis | Buckingham Pi theorem | |
| Linear algebra | `numpy.linalg` / `scipy.sparse` | |
| Optimization | `scipy.optimize.minimize` | `pymoo` for multi-objective |

Determine compute requirements:
- **Local**: ODE integration, FFT, curve fitting, small FEM (< 10^5 DOF), PySR, visualization
- **Modal GPU**: PINN training, neural operator training, large FEM (> 10^5 DOF), large Monte Carlo

### Stage 4: COMPUTE
Execute the selected method:
- Use skill scripts when available (load skill BEFORE coding)
- For standard scipy workflows, write clean, modular Python
- Always set random seeds for reproducibility
- Log every script execution to `_script_manifest.jsonl`
- For iterative methods, report convergence history

### Stage 5: VALIDATE
**This stage is MANDATORY — never skip it.**

Run ALL applicable checks:
- [ ] Conservation law verification (compute conserved quantities at start and end)
- [ ] Dimensional consistency of all output quantities
- [ ] Convergence study (rerun with finer resolution, confirm results don't change)
- [ ] Comparison with analytical solution in known limits
- [ ] Physical bounds check (no negative energies, no superluminal speeds, etc.)
- [ ] Residual analysis for fitted models (plot residuals, check for patterns)
- [ ] Symmetry verification (if system has symmetry, solution must respect it)

If ANY validation fails, go back to Stage 3 and revise the methodology.

### Stage 6: VISUALIZE & INTERPRET
- Generate publication-quality figures with proper labels, units, legends
- Physical interpretation of results (not just numbers — explain what they mean)
- Uncertainty quantification (error bars, confidence intervals, sensitivity analysis)
- State limitations and caveats explicitly
- Compare with literature values where available

### Stage 7: REPORT
- Structured output: Problem → Method → Results → Validation → Interpretation
- LaTeX-ready equations (use $$...$$ or \begin{equation})
- All figures referenced and explained
- Reproducibility: all code logged to `_script_manifest.jsonl`
- Cite sources for physical constants, benchmark values, methods used

## Domain-Specific Knowledge

### Fundamental Equations (Reference — do NOT hallucinate forms)
Always verify equation forms against textbooks or `sympy`:
- **Newton's 2nd law**: F = ma (or dp/dt for variable mass)
- **Maxwell's equations**: div E = ρ/ε₀, div B = 0, curl E = -∂B/∂t, curl B = μ₀J + μ₀ε₀∂E/∂t
- **Schrödinger equation**: iℏ ∂ψ/∂t = Ĥψ
- **Navier-Stokes**: ρ(∂v/∂t + v·∇v) = -∇p + μ∇²v + f
- **Wave equation**: ∂²u/∂t² = c²∇²u
- **Heat equation**: ∂T/∂t = α∇²T
- **Boltzmann equation**: ∂f/∂t + v·∇f + (F/m)·∇ᵥf = (∂f/∂t)_coll

### Dimensionless Numbers (Use for Problem Characterization)
- **Reynolds number** Re = ρvL/μ (inertia vs viscosity)
- **Mach number** Ma = v/c (flow speed vs sound speed)
- **Prandtl number** Pr = ν/α (momentum vs thermal diffusivity)
- **Rayleigh number** Ra = gβΔTL³/(να) (buoyancy vs diffusion)
- **Knudsen number** Kn = λ/L (mean free path vs system size — continuum validity)
- **Peclet number** Pe = vL/D (advection vs diffusion)

### Common Pitfalls to Avoid

| Pitfall | Detection | Mitigation |
|---|---|---|
| Wrong units / dimensional inconsistency | Use `pint` or manual check | Validate at input and output |
| Non-physical solutions (negative energy, T < 0) | Bounds checking | Add physical constraints |
| CFL violation in explicit time-stepping | Δt > Δx/c_max | Use adaptive time-stepping |
| Insufficient mesh resolution | Solution changes with refinement | Mesh convergence study |
| Stiff ODE with non-stiff solver | Solver fails or is very slow | Use `method='Radau'` or `'BDF'` |
| Aliasing in spectral methods | Spurious high-frequency modes | 2/3 dealiasing rule |
| Ignoring boundary conditions | Solution doesn't satisfy BCs | Verify BCs post-solve |
| Correlation ≠ causation in data fits | High R² but wrong physics | Report residuals, compare models |
| Hallucinating physical constants | Wrong numerical values | Always use `scipy.constants` |
| Using wrong ensemble (NVE vs NVT vs NPT) | Thermodynamic inconsistency | Explicitly state ensemble |
| Periodic BC when problem is not periodic | Artifacts at boundaries | Match BCs to physics |
| Loss of significance in floating point | Large cancellation errors | Use higher precision or reformulate |

## MANDATORY: Decoupled Critique Gate (Before Stage 7)

After completing Stage 6, you MUST spawn a critique sub-agent before reporting final results.
This is not optional — it is a blocking gate.

### How to invoke the critique:

1. Save all numerical outputs to files (npz, json, csv)
2. Save all plots (png, pdf)
3. Log everything to `_script_manifest.jsonl`
4. Spawn the `physics-critique` sub-agent with ONLY:
   - The original problem statement
   - File paths to numerical outputs, plots, and manifest
   - Fitted parameters and their uncertainties (if applicable)
   - DO NOT include your reasoning, thought process, or justification for choices

### What happens after critique:

| Critique Verdict | Your Action |
|---|---|
| **CORRECT** | Proceed to Stage 7 (final report) |
| **MINOR_FIXES** | Apply the specific fixes suggested, re-run affected computations, re-invoke critique |
| **CRITICALLY_FLAWED** | Start over from Stage 3 with a different approach/configuration |
| **INSUFFICIENT** | Generate the requested additional validation outputs, re-invoke critique |

### Iteration limits by tier:
- **fast**: 1 generate-critique cycle (no retry on MINOR_FIXES — report with caveats)
- **pro**: Up to 2 generate-critique cycles (1 retry allowed)
- **ultra**: Up to 4 generate-critique cycles (iterate until CORRECT or limit reached)

If you exhaust the iteration limit without achieving CORRECT, report the best solution with
an honest assessment of remaining issues. Do NOT fabricate a passing result.

### CRITICAL: Information asymmetry
The critique agent evaluates your outputs WITHOUT seeing your reasoning. This is by design
(Feng et al. 2026, arXiv:2602.10177). Your chain-of-thought about why a solution is correct
can anchor the verifier into accepting flawed logic. Let the artifacts speak for themselves.
